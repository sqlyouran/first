# 编码规约

三条底线，不可违反：

## 1. YAGNI — You Aren't Gonna Need It
- 只写当前 spec 需要的代码。
- 不预留"将来可能用到"的参数、接口、抽象。
- 不重构未被要求修改的代码。

## 2. DRY — Don't Repeat Yourself
- 出现第三次相同模式才抽。第一次照写，第二次容忍，第三次提炼。
- 不要为了消除两处微小重复而引入新抽象层。

## 3. TDD — Test-Driven Development
- 严格遵守 RED → GREEN → REFACTOR：
  1. 先写一个**会失败**的测试，亲眼看它失败。
  2. 写**最小**实现让它通过。
  3. 在绿灯下重构。
- 实现前写的代码若没对应失败测试，必须删掉重来。

## 注释
- 仅在意图无法从代码本身看出时才加注释。
- 优先短注释，避免大段块注释。
- 不写"这是干什么的"流水账，写"为什么这么干"。
