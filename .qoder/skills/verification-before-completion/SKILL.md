# verification-before-completion

**信条**：证据高于声明。在你说"完成"之前，先给出可验证的证据。

## 必备证据

每次声称完成前，必须能贴出：

1. **测试命令 + 输出**
   ```
   $ pytest tests/test_foo.py -v
   ...
   ===== 4 passed in 0.12s =====
   ```

2. **关键路径手动验证**（如果是用户可见行为）
   - 实际运行命令、调用 API、点界面。
   - 贴出 stdout / 截图 / curl 输出。

3. **回归确认**：跑一遍**全量**测试，不是只跑你新写的。

## 反模式（高危）

- "应该没问题" / "理论上 OK" / "看起来对" → 都是没验证。
- 只跑了新增测试，没跑全量 → 可能破坏了别处。
- 测试通过了但功能没真的运行起来一次 → 测试可能在错的层级。
- 把"我改了代码"等同于"问题修好了"。

## Bug 修复的额外验证

- 在改动前**先复现 bug**——写一个失败测试。
- 改完后这个测试必须通过。
- 这个测试要永远留下来作为回归保护。

## 一句话

**没跑过的代码不算写过，没失败过的测试不算保护。**
